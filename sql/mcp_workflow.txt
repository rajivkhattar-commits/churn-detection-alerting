Using AskWoltAI schema in production (no MCP required on the API host)
======================================================================

Canonical table and join documentation lives in the AskWoltAI repo file:
  ask-wolt-mcp/schema.py  →  variable SCHEMA_DESCRIPTION

The churn API loads that file when ASKWOLT_MCP_HOME points to the clone (same content
MCP tool get_schema is built from). You do NOT need Cursor MCP online in production.

Workflow
--------
1. Keep ask-wolt-mcp cloned on servers or in the build image.
2. Set ASKWOLT_MCP_HOME=/path/to/ask-wolt-mcp on the churn API.
3. Optional: CANONICAL_JOINS_JSON for deployment-specific structured overrides.
4. Optional: ENTERPRISE_DEFINITION_JSON for ENT merchant prose.

Cursor MCP (AskWoltAI + Snowflake) is for interactive discovery and SQL validation
while you build dbt models or feature pipelines — not a runtime dependency for this API.

Churn analytics (schema.py)
---------------------------
The AskWoltAI repo documents pre-aggregated churn on PRODUCTION.MERCHANT.VENUE_COUNTRY_REPORTING_MONTHLY,
_WEEKLY, and _DAILY (aligned columns; choose grain by question). Monthly is for portfolio churn rate and
Looker-style KPIs; daily adds venue-level rows with CHURN_DATE and delete/unpublish reason fields. The API’s
schema excerpt (see WOLT_SCHEMA_EXCERPT_CHARS) may not include those sections — search SCHEMA_DESCRIPTION in
schema.py for "CHURN / VENUE LIFECYCLE" when building cohort or label queries.
